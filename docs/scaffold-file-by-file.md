# RaptorFlow Scaffold: File-by-File Guide

This document explains the production-grade scaffold in this repository, file by file.

The goal is not to restate source code line by line. The goal is to answer four questions for every scaffold file:

1. What is this file for?
2. Why does it exist in this phase?
3. What should eventually go in it?
4. What should not go in it?

## Scope

This guide covers the source scaffold and the source corpus that informed it.

It intentionally excludes generated or cache artifacts:

- `.git/`
- `node_modules/`
- `.next/`
- `target/`
- `dist/`
- `.turbo/`
- `*.tsbuildinfo`

## How To Read This

- `Purpose` explains why the file exists.
- `What goes here` explains the intended long-term contents.
- `Do not put here` explains common boundary mistakes.

## Root Workspace Files

- `.editorconfig`

  - Purpose: repository-wide editor defaults.
  - What goes here: indentation, line endings, charset, trailing newline policy.
  - Do not put here: formatter rules that belong to Prettier or ESLint.

- `.env.example`

  - Purpose: local-development environment variable template.
  - What goes here: non-secret example values, required variable names, local defaults.
  - Do not put here: real secrets, production credentials, anything copied out of Secrets Manager.

- `.gitattributes`

  - Purpose: Git normalization and file handling rules.
  - What goes here: line-ending normalization, binary-file declarations, diff handling hints.
  - Do not put here: branch policy or CI logic.

- `.gitignore`

  - Purpose: ignore generated files and local-only state.
  - What goes here: caches, builds, local env files, IDE noise, package manager stores.
  - Do not put here: tracked source directories or shared fixtures.

- `.npmrc`

  - Purpose: repo-level npm/pnpm behavior.
  - What goes here: workspace install behavior, lockfile discipline, dependency policy.
  - Do not put here: tokens or registry secrets.

- `.nvmrc`

  - Purpose: Node version pin for humans and tooling.
  - What goes here: one Node major/minor target used by the repo.
  - Do not put here: pnpm version or shell logic.

- `.prettierignore`

  - Purpose: exclude paths from Prettier.
  - What goes here: generated files, lockfiles only if intentionally excluded, vendor-like assets.
  - Do not put here: normal source files that should still be formatted.

- `.prettierrc.json`

  - Purpose: formatting rules for JS/TS/JSON/Markdown.
  - What goes here: shared formatting decisions.
  - Do not put here: lint rules or semantic code policy.

- `Cargo.lock`

  - Purpose: resolved Rust dependency lockfile.
  - What goes here: Cargo-generated dependency resolution state.
  - Do not put here: manual edits except in exceptional dependency repair work.

- `Cargo.toml`

  - Purpose: Rust workspace definition and shared dependency versions.
  - What goes here: workspace members, workspace-wide metadata, shared crate dependency pins.
  - Do not put here: app runtime code or SQLx queries.

- `clippy.toml`

  - Purpose: Rust lint policy overrides.
  - What goes here: Clippy thresholds and exceptions justified across the workspace.
  - Do not put here: build or rustfmt configuration.

- `CONTRIBUTING.md`

  - Purpose: contributor workflow reference.
  - What goes here: branch flow, validation expectations, commit/PR standards, repo conventions.
  - Do not put here: architecture truth that belongs in canonical docs.

- `docker-compose.yml`

  - Purpose: local stack orchestration.
  - What goes here: web, API, Postgres, PgBouncer, Qdrant, Dragonfly, volumes, network wiring.
  - Do not put here: production deployment assumptions that only belong in ECS/OpenTofu.

- `package.json`

  - Purpose: monorepo Node entrypoint.
  - What goes here: top-level scripts, package manager pin, workspace-wide dev tooling.
  - Do not put here: app-specific dependencies that belong in workspace packages.

- `pnpm-lock.yaml`

  - Purpose: resolved Node dependency lockfile.
  - What goes here: exact dependency graph generated by pnpm.
  - Do not put here: hand-written configuration.

- `pnpm-workspace.yaml`

  - Purpose: pnpm workspace membership.
  - What goes here: workspace package globs.
  - Do not put here: script logic or dependency versions.

- `README.md`

  - Purpose: repo entrypoint.
  - What goes here: stack summary, layout, bootstrap commands, high-level navigation.
  - Do not put here: deep subsystem design; link to docs instead.

- `rust-toolchain.toml`

  - Purpose: Rust toolchain pin.
  - What goes here: Rust version and optionally components/targets.
  - Do not put here: dependency configuration.

- `SECURITY.md`

  - Purpose: security reporting and handling policy.
  - What goes here: disclosure path, sensitive-data expectations, response flow.
  - Do not put here: the threat model itself.

- `tsconfig.base.json`

  - Purpose: shared TypeScript compiler settings.
  - What goes here: strictness, module settings, path aliases, base compiler options.
  - Do not put here: app-specific emit behavior that belongs in package-local configs.

- `turbo.json`
  - Purpose: Turborepo task graph and cache policy.
  - What goes here: build/lint/typecheck/test task dependencies, outputs, global inputs.
  - Do not put here: package-specific scripts that belong in package manifests.

## Practical Ownership Rules

Use these rules to decide where new work should go:

- If the change defines a shared shape used by multiple systems, start in `crates/contracts`, `packages/contracts`, or `schemas`.
- If the change defines a route surface, start in the appropriate backend domain crate and mirror it in `schemas/openapi/api-v1.yaml`.
- If the change defines a frontend route, put the page in `apps/web/src/app/...` and shared presentation in `apps/web/src/components/...`.
- If the change defines durable state, add it in `database/migrations` first, then reflect it in shared contracts.
- If the change defines cloud infrastructure, put it in `infra/tofu/modules/*` and wire it from `infra/tofu/environments/*`.
- If the change exists only to explain a system boundary or preserve a source decision, put it in `docs/canonical`, `docs/adrs`, `docs/runbooks`, or `docs/source-digests` depending on the purpose.

## What This Scaffold Still Intentionally Does Not Do

This repository is still scaffold-first.

That means many files define:

- boundaries
- contracts
- routes
- schemas
- storage
- deployment surfaces
- validation checks

But they still intentionally defer:

- final business logic
- final prompt prose
- full scraping/browser execution
- real Office rendering assets and motion
- production-grade worker internals
- live data mutations beyond stubs

That is deliberate. The scaffold is the frame the system will be built inside.
